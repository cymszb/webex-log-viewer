Three Approaches to Multi-User Deployment

Shared topics config · Per-user log uploads · Session-cookie isolation

A

Server-Side Sessions (Recommended)

Flask server sessions (signed cookies) identify each user. Uploaded log files are stored temporarily on the server (e.g. /tmp/sessions/<session_id>/log.txt), parsed on demand, and cleaned up after a TTL. topics.json remains a shared flat file with a file lock for safe concurrent writes. Drag & drop is removed — all loading goes through upload.

Pros

  • Minimal new dependencies (flask-session or just signed cookies)
  • No database needed
  • Straightforward to deploy (gunicorn + nginx)
  • Clean separation: server owns log data, client owns nothing sensitive

Cons

  • Large log files consume server disk temporarily
  • Need TTL cleanup job for orphaned sessions
B

Keep Client-Side Parsing + Add Upload API

Keep the existing drag-&-drop client-side parser. Add a new POST /api/upload endpoint for server-side upload. Parsed log JSON is stored in the Flask session (in-memory or signed cookie). No files written to disk beyond the existing current_log.txt.

Pros

  • Least disruption to existing frontend
  • No disk usage for uploaded files

Cons

  • Large logs (100k+ lines) bloat session storage
  • Duplicates parsing logic (Python + JS)
  • Cookie size limits can be a problem
C

Full Rewrite with a Proper DB

Replace topics.json with SQLite (or Postgres). Add user sessions table. Store uploaded log chunks in the DB. Full REST API redesign.

Pros

  • Most scalable long-term
  • Concurrent write safety built-in

Cons

  • Significant rewrite — high risk, slow to deliver
  • Overkill for a small team tool
  • Loses the "no dependencies" simplicity
